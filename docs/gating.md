# 軌跡錄製的定位點 Gating 規則

本文件規範軌跡錄製時, 一個 GPS fix 要通過哪些檢驗才能寫入軌跡檔 (下稱 **gating**).
本文僅定義規則與設計理由, 不含程式實作. (定案於 2026-07-02 的討論)

## 1. 背景與動機

登山軌跡的定位雜訊高度集中在**靜止與低速時段** (休息, 拍照, 排隊拉繩): 位置在
accuracy 圈內隨機遊走, 產生俗稱 "鳥巢" 的原地亂線. 其後果不只是難看 - 已走距離被
灌水, 平均速度失真, 經 DEM 查得的爬升也跟著抖動 (水平誤差在陡坡上直接轉成垂直誤差).

處理方式的選擇: 這個問題的本質是 "**該不該收這個點**" (gating), 而不是 "把收進來的
點修得更準" (smoothing). 因此:

- **不採用 Kalman filter** - GNSS 晶片與 Android 定位堆疊本來就在跑 Kalman, 收到的
  fix 已是濾波輸出; 再疊一層對登山動態 (之字形急轉) 只會磨圓彎角, 低速時又引入
  落後. 若日後實測 (gating 版本錄製的真實行程) 仍顯示明顯抖動, 再考慮於顯示 / 匯出
  層做後處理, 不在寫檔前改寫量測值.
- **不採用 Douglas-Peucker** - DP 是簡化演算法, 原理是保留 "偏離基線最遠的點", 而
  漂移尖刺正是偏離最遠的點; 對去噪是反效果. 其正確用途 (匯出減點, 長軌跡渲染) 與
  本問題正交.
- **不採用移動平均 / 指數平滑** - 與 Kalman 同樣磨彎角, 但無 accuracy 加權與運動
  模型, 是更粗糙的版本.

## 2. 設計原則

1. **寫入的都是真實量測** - gating 只決定收或不收, 絕不改寫座標. 軌跡檔內每個點都
   是定位系統實際回報過的位置.
2. **Gate 設在錄製路徑, 不設在顯示路徑** - 每個 fix 進來後走兩條互相獨立的路,
   gate 只設在錄製那一條:

   ```
   GpsSource.fix (每個原始 fix)
    ├─ 顯示路徑: 現在位置標記, 精度圈, 過期轉灰, 畫面跟隨 / 歸位   <- 不過濾
    └─ 錄製路徑: Recorder.onFix() -> 軌跡檔 + 紅色軌跡線           <- gating 在此
   ```

   這**不是** "螢幕顯示與記錄內容不一致". 紅色軌跡線與軌跡檔是同一份受 gate 的
   資料, 畫面上的紅線就是檔案裡的軌跡, 一點不差. 與檔案不同步的只有**現在位置
   標記** - 它本來就不是記錄, 而是儀表: 誠實呈現定位系統此刻的原始回報. 被 gate
   掉的 fix 照常移動標記 (靜止時的飄動配上精度圈, 本身就是訊號品質的可視化),
   照常重置過期計時 (它證明衛星訊號還活著; 否則 "被 gate" 與 "訊號消失" 在轉灰
   邏輯眼中無法區分), 也照常參與畫面跟隨 / 歸位判斷.

   標記與紅線前端因此會在靜止漂移時短暫分離. 這個分離是誠實的: GPS 真的在飄, 但
   人真的沒有移動, 軌跡如實記下 "沒有移動". 全畫面唯一 "看得到但檔案裡沒有" 的
   元素是橡皮筋段 (§6), 它是純視覺補綴, 不代表任何記錄內容.
3. **不假設移動方式** - 規則不得內建 "使用者一定在步行" 的假設; 在火車, 接駁車,
   纜車上錄製都必須正確運作 (見 §5).
4. **網路定位不進軌跡** - 錄製模式只訂閱 GPS provider, gating 的輸入永遠是衛星
   fix. 濾波救不回 cell / WiFi fix 的系統性偏置, 這不是 gating 要解的問題.

## 3. 規則

前置條件 (既有行為, 非本文新增): 只有 `RECORDING` 狀態評估以下規則; `PAUSED` 時
fix 一律丟棄. "錨點" 指**最後一個已寫入軌跡的點** (續錄時為原軌跡的最後一點).
Session 的第一個點沒有錨點, 只受規則 1 檢驗, 通過即寫入.

依序套用, 任一規則不通過即丟棄 (規則 2 為擱置待佐證):

### 規則 1: Accuracy 門檻

`accuracyMeters > ACCURACY_MAX_M` 的 fix 不寫入 - 定位系統自己都說不準的點, 不值得
成為軌跡的一部分. `accuracyMeters` 為 null (罕見, 裝置未回報) 時視同通過, 交由後續
規則把關; 全數拒收會讓少數裝置完全無法錄製.

### 規則 2: 位移與 Doppler 速度的自洽檢驗 (跳點防護)

計算相對錨點的隱含速度 `vDisp = distance(錨點, fix) / dt` (dt 取兩點 `timeMs` 差).
GNSS 晶片以 Doppler 頻移量得的瞬時速度 `vDop` (= `speedMps`) 與位置誤差幾乎獨立,
可作交叉檢驗:

- **自洽** (`vDisp <= max(K_DOP * vDop, V_FLOOR)`): 位移與實際速度相符, 進入規則 3.
- **不自洽** (超過上式): 這個位移未獲速度佐證, 可能是尖刺. 將此 fix **擱置**
  (pending), 等下一個 fix:
  - 下一個 fix 以擱置點為錨通過同樣的自洽檢驗 (運動確實從擱置點延續) -> 佐證成立,
    寫入擱置點, 再以擱置點為錨評估新 fix;
  - 否則 (跳回原地) -> 擱置點是尖刺, 丟棄, 新 fix 改以原錨點評估.
- `vDop` 為 null 時, 門檻退化為 `V_FLOOR` (等同兩拍佐證機制).

擱置點至多一個; 錄製暫停, 結束, 或 fix 過期 (fixStale 轉 true) 時清空.

### 規則 3: Adaptive 間距門檻

`distance(錨點, fix) < clamp(K_ACC * accuracyMeters, SPACING_MIN_M, SPACING_MAX_M)`
的 fix 不寫入 - 位移沒有真正走出量測不確定性的範圍, 就不視為移動. 這一條直接消滅
靜止時的鳥巢, 也是距離統計灌水的主要解方. `accuracyMeters` 為 null 時門檻取
`SPACING_MIN_M`.

取代自 GpsSource 訂閱移除的 OS 層 minDistance 過濾 (該值必須為 0, 否則靜止與訊號
消失在過期偵測眼中無法區分, 見 GpsSource 的註解).

## 4. 參數

| 參數 | 值 | 意義 |
|---|---|---|
| `ACCURACY_MAX_M` | 30 m | 規則 1: accuracy 超過此值的 fix 不寫入 |
| `K_DOP` | 3 | 規則 2: 隱含速度容許為 Doppler 速度的倍數 |
| `V_FLOOR` | 10 m/s | 規則 2: 自洽門檻下限 (低速時 Doppler 噪聲的保護墊) |
| `K_ACC` | 1.0 | 規則 3: 間距門檻對 accuracy 的比例 |
| `SPACING_MIN_M` | 5 m | 規則 3: 間距門檻下限 |
| `SPACING_MAX_M` | 30 m | 規則 3: 間距門檻上限 (與 `ACCURACY_MAX_M` 一致) |

所有數值為保守預設, **待真實行程的錄製資料回來後校準**; 調整時同步更新本表.

## 5. 邊界情境驗證

- **火車 / 電車 / 纜車**: vDisp 約 30 m/s, vDop 亦約 30 m/s -> 自洽, 直接通過;
  每秒位移遠超間距門檻, 逐點寫入. 若規則 2 設計成硬性速度上限 (如 10 m/s), 軌跡
  會從上車起完全凍結且永不恢復 - 這正是採用 Doppler 自洽而非速度上限的原因.
- **靜止漂移**: 位移多半困在間距門檻內 (規則 3 擋下); 偶發的大幅尖刺 vDisp 高而
  vDop 近零 -> 不自洽, 且下一個 fix 跳回原地, 佐證失敗丟棄 (規則 2 擋下).
- **隧道 / 峽谷出口的真實跳躍**: 長時間無 fix 後, dt 大使 vDisp 攤平 (500 m / 600 s
  = 0.8 m/s), 通常直接自洽; 即使不自洽, 下一個 fix 延續新位置, 佐證成立, 僅延遲
  一拍入檔.
- **原地繞小圈**: 位移始終小於間距門檻時記成一個點 - 所有間距門檻的共同行為, 5 m
  尺度下對登山軌跡可忽略.
- **標記與紅線前端的分離**: gating 使前端最多落後標記一個間距門檻值 (5-30 m,
  與 GPS 誤差同數量級). 移動時幾秒內即追平; 靜止漂移時 "不跟上" 正是目的. 視覺
  連續性由顯示層配套解決 (§6). 畫面跟隨 / 歸位判斷不受影響 - 該邏輯只看
  `GpsSource.fix` (標記位置), 不看軌跡前端.

## 6. 顯示層配套: 橡皮筋段 (前端補接線段)

Gating 造成一個純視覺的問題: 紅線只畫到最後一個**寫入**的點, 而標記跟著**每個原始
fix** 走, 於是靜止漂移時兩者之間會出現一段空隙 (最大為間距門檻, 5-30 m), 看起來像
紅線斷在半路, 沒有連到人的位置.

解法是在這個空隙補畫一段線, 一端固定在紅線前端 (最後寫入點), 另一端綁在標記上:

```
        已寫入的軌跡 (= 檔案內容)         橡皮筋段 (僅畫面, 不入檔)
        ●━━━━●━━━━●━━━━●╌╌╌╌╌╌╌╌◉  <- 現在位置標記
                       ^                    (跟著每個原始 fix 移動)
                   最後寫入點
```

標記飄到哪裡, 這段線就跟著拉伸到哪裡; 標記飄回來, 它就縮短 - 故稱橡皮筋. 兩種
情境下的行為:

- **正常移動時**: 位移一超過間距門檻, 新點就寫入, 紅線前端追上標記, 橡皮筋段縮到
  幾乎看不見 - 與沒有 gating 時的畫面無異.
- **靜止漂移時**: 紅線前端停在原地 (這正是 gating 的目的), 橡皮筋段跟著標記輕微
  擺動, 畫面上紅線始終連到標記, 不會出現斷頭.

效果: 檔案裡是 gate 過的乾淨軌跡, 畫面上軌跡與標記永不斷開.

## 7. 實作位置

- Gating 全部位於 `Recorder.onFix()` - 它持有錨點 (最後寫入點) 與錄製狀態, 是純
  Kotlin, 三條規則可用合成漂移資料寫單元測試. `onFix()` 簽名需擴充
  `accuracyMeters: Float?` 與 `speedMps: Float?`, `RecordingService` 呼叫端跟著多傳.
- 即時 polyline (`Recorder.points`) 與檔案寫入同步受 gate, 畫面紅線與檔案內容一致.
- **不得**放在 `GpsSource` 或 `CurrentLocationLayer`: gating 的判斷依據是錄製 session
  的狀態 (錨點), 且顯示端不該被 gate (§2 原則 2); 放上游會讓被拒的 fix 連標記,
  過期偵測, 跟隨判斷一起消失, 依賴方向也反了.
- 橡皮筋段位於 `RecordedTrackLayer` (顯示層).
