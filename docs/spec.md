# 登山地圖 App 規格書 v0.1

> 本文件為起始規格, 預期會在開發過程中迭代.

---

## 0. 願景

打造一款解決現有登山 App "**UI/操作體驗**" 與 "**軌跡與路線規劃**" 兩大痛點的 Android 離線地圖 App. 僅做 Android, 不考慮 iOS.

定位: 自用為主, 檔案分享導向, 不依賴後端服務的個人化登山工具. 圖資不自行生產, 直接取用 RudyMap (MOI.OSM Taiwan TOPO) 的 mapsforge `.map` 底圖, 其隨附樣式, 與 `.hgt` DEM, 由 App 從 RudyMap 公開鏡像下載, 僅供自用, 不轉存散布.

---

## 1. 需求

### 1.1 功能需求 (v1)

| 編號 | 需求 | 細節 |
|------|------|------|
| F1 | 離線地圖瀏覽 | 完全離線可用, mapsforge 向量圖磚, 可連續捏放縮放與旋轉 (標準渲染器, 無 3D 傾斜俯角) |
| F2 | 自動路線規劃 | 給起終點 + 中途 waypoint, snap 到 OSM 山徑 |
| F3 | 路線量測 | 規劃完顯示總長度, 總爬升/下降, 海拔剖面圖 |
| F4 | GPS 軌跡錄製 | 背景錄製, 螢幕關閉持續記錄 |
| F5 | 軌跡即時統計 | 已走距離, 總爬升, 時間, 平均速度 |
| F6 | 即時海拔剖面 | 錄製中顯示今日路程已走剖面 |
| F7 | GPX/KML 匯入匯出 | tracks + routes + waypoints, 相容主流登山 App |
| F8 | 圖資下載與更新 | App 內直接從 RudyMap 鏡像下載 `.map` + DEM + 樣式 (需連網); 設定頁 "檢查更新" 比對 HTTP `Last-Modified` 判斷新版; 並保留 "手動匯入本地檔" 備援 |
| F9 | Google Maps 跳轉 | "在 Google Maps 開啟此位置" 按鈕 |

### 1.2 非功能需求

- 平台: Android 8.0+ (僅 Android, 不做 iOS)
- 螢幕方向: 地圖類畫面 (地圖檢視/規劃/錄製主畫面/海拔剖面/軌跡地圖) 支援直向與橫向自適應 (responsive layout); 設定/清單類維持直向. 錄製中轉向 (configuration change) 不得中斷前景服務與 GPS 記錄
- 離線範圍: 台灣全島 (RudyMap 圖資涵蓋範圍; 無多區域/海外擴充)
- 啟動時間目標: < 3 秒
- 地圖載入: < 1 秒到首幀
- GPS 取樣率: 可配置, 預設 5 秒
- 電量目標: 連續錄製 8 小時 < 30% 電量消耗
- App 安裝體積: < 100 MB (RudyMap 圖資由 App 首次啟動下載, 不隨包; 底圖本身約 298 MB 壓縮檔)

### 1.3 v1 明確排除 (v2 候選清單)

- ❌ 林務局通訊點/避難點疊圖 (**強烈建議盡早補上, 救命用**)
- ❌ 自己手動標記私人 POI (水源, 營地, 危險點)
- ❌ 衛星/航照影像疊圖 (原本選了, 但 Google Maps 改跳轉後可以延後)
- ❌ 多日縱走規劃 (含過夜點/每日行程)
- ❌ 偏離預定路線警示
- ❌ 多裝置同步, 雲端帳號

## 2. 技術架構

### 2.1 技術棧決策

| 層級 | 選擇 | 為何 |
|------|------|------|
| App 框架 | **Flutter 3.x** (僅 Android) | UI 自訂彈性高, 單一 Dart 程式碼庫; 不再以 iOS 重用為理由 |
| 地圖渲染 | **`mapsforge_flutter`** (純 Dart, mapsforge 標準渲染器) | 直接吃 RudyMap `.map`; 在 Flutter 行程內, 地圖不必走 platform channel; OruxMaps 證實標準渲染器可達連續捏放與旋轉 |
| 圖磚格式 | **mapsforge `.map`** (取自 RudyMap) | 直接使用 RudyMap 既有圖資, 不自行生產; 台灣為單一整檔 |
| 地圖樣式 | **RudyMap 隨附 render theme** (原樣使用) | 不自製美術, 連深色版都現成 |
| 路由引擎 | **GraphHopper** (Java, embed 進 Android) | 完全離線, 支援 hiking profile, 行動端整合相對成熟 |
| 高程處理 | **RudyMap `.hgt` DEM** (`hgtmix`) | App 端查詢海拔; mapsforge 可由 `.hgt` 即時算 hillshade, 不自行預烘 |
| 等高線 | **內嵌於 RudyMap `.map`** (TOPO 圖, Phase 0 驗證) | 由樣式渲染, 不自行產出 |
| 本地儲存 | **SQLite** (軌跡/路線/設定) + 檔案 (`.map`/路由圖/DEM) | 標準作法 |
| GPS 服務 | **Android Foreground Service** + Wake Lock | 唯一可靠的背景錄製方式 |
| Backend | **無** | 不託管任何資料; 圖資由 App 直接抓 RudyMap 公開鏡像 (靜態 HTTPS) |

### 2.3 圖資取得與更新

**不自行生產底圖**. 底圖/樣式/DEM 全部來自 RudyMap, 由 App 直接下載:

```
RudyMap 公開鏡像 (擇一, 可容錯輪替)
  https://rudymap.tw/v1/
  https://moi.kcwu.csie.org/
  https://map.happyman.idv.tw/rudy/
    │
    ├─ MOI_OSM_Taiwan_TOPO_Rudy.map.zip   (~298MB; Lite ~168MB) -> 底圖
    ├─ hgtmix.zip (~46MB, 30m) / hgt90.zip (~8MB)               -> DEM (.hgt)
    └─ MOI_OSM_Taiwan_TOPO_Rudy_hs_style.zip (~0.4MB)           -> 樣式 (含深色)
```

- 皆為**靜態 HTTP GET, 無表單/JS**; 檔名固定, 內容每週四更新 (beta 週一三六).
- **無 manifest, 無 SHA256**. 版本日期 (`vYYYY.MM.DD`) 只在網頁, 不在檔名.

唯一仍需自行產出的是**路由圖**: 取 OSM Taiwan extract (Geofabrik) 經 GraphHopper import (hiking profile) 產生路由圖目錄. 可離線預先產好放入裝置, 或日後評估能否在裝置上建. (與底圖無關, 不依賴 RudyMap.)

App 端 "檢查更新" 流程:
1. 對各檔案 URL 發 **HTTP HEAD**, 取 `Last-Modified` / `ETag` / `Content-Length`.
2. 與本地上次下載時記錄的值比對, 有變即列為可更新 (顯示名稱, 大小).
3. 使用者選擇要更新的項目 (底圖 / DEM / 樣式).
4. 下載 zip 到暫存目錄 -> 解壓 -> 完整性檢查 (能否正常開啟; 無官方雜湊可比) -> 通過才原子切換 (atomic swap).
5. 多鏡像容錯: 某鏡像失敗時自動改用次一鏡像.

備援: 若鏡像不可達或使用者已有檔案, 可由設定頁 "手動匯入本地 `.map`/DEM/樣式".

## 4. 主要使用者流程

### 4.1 規劃一條路線

1. 開啟 App -> 主畫面顯示台灣登山地圖 (或上次瀏覽位置)
2. 點選工具列 "規劃" 進入規劃模式
3. **長按地圖** -> 選 "設為起點"
4. **再長按** -> 選 "設為終點", App 自動算路線
5. **拖曳路線中段** -> 自動新增 waypoint 強制經過
6. **長按 waypoint 拖曳** -> 移動位置; 雙擊刪除
7. 下方面板顯示: 總長度, 總爬升/下降, 預計時間, 海拔剖面圖
8. 滑動海拔剖面 -> 地圖上同步顯示對應位置標記
9. "儲存" -> 命名並存進路線庫
10. "匯出 GPX" -> 透過 Android Share Sheet 分享

### 4.2 上山錄製軌跡

1. (上山前) 開啟 App -> "載入路線" (選擇性)
2. 主畫面右下 "●錄製" 按鈕 (永遠常駐, 單手可按)
3. 確認權限後開始錄製: 通知列常駐通知 (Foreground Service)
4. 螢幕可關, 放口袋; GPS 持續錄
5. 主畫面浮動 widget: 距離 | 爬升 | 時間 | 速度
6. 下緣可上滑出小型海拔剖面 (顯示已走路程)
7. "定位" 按鈕: 自動置中到當前位置; 再按一次切換 "跟隨模式" (地圖跟著走)
8. "結束" -> 確認後自動儲存到軌跡庫
9. 軌跡頁可查看: 完整地圖, 剖面, 統計, 並匯出 GPX

### 4.3 圖資更新

1. 設定 -> 圖資管理 -> "檢查更新" (需連網)
2. App 對 RudyMap 鏡像各檔發 HTTP HEAD, 比對本地記錄的 `Last-Modified`
3. 列出有更新的項目與大小 (底圖 ~298MB, DEM ~46MB, 樣式 ~0.4MB)
4. 全選 / 個別選擇 -> "下載更新" (鏡像失敗自動換下一個)
5. 下載中可背景進行 (不阻擋 App 使用)
6. 解壓 + 完整性檢查通過後, 原子切換到新版本
7. 提示 "更新完成, 重新載入地圖"
8. (備援) 無法連網時可改 "手動匯入本地檔"

## 5. UI/UX 設計原則

針對你列的痛點, 這些是 v1 必須做對的地方:

| 原則 | 具體做法 |
|------|---------|
| 單手操作優先 | 重要按鈕 (錄製, 定位, 選單) 放下半螢幕, 拇指區 |
| 手套友善 | 所有可點按元件最小 48dp, 間距充足 |
| 一鍵錄製 | 主畫面常駐錄製按鈕, 不必進選單 |
| 海拔剖面互動 | 滑動剖面時地圖同步定位 |
| 直橫向自適應 | 地圖類畫面隨手機方向自動重排; 橫向利於看剖面與寬地圖. 錄製中轉向不中斷記錄 |
| 中英文切換 | 一鍵切換圖資語言 (`name:zh` <-> `name:en`) - **取決於 RudyMap `.map` 是否含 `name:en`, Phase 0 驗證; 若無則此項取消** |
| 縮放手勢 | 雙指捏放, 雙指旋轉, 雙擊放大, 雙指拍擊縮小 |

**地圖圖面美術 (顏色, 線寬, 等高線, 符號) 不自製, 一律沿用 RudyMap 隨附 render theme.** 以上原則只規範 App 介面外殼 (按鈕配置, 手勢, 互動), 不含地圖畫布樣式.

註: RudyMap 樣式本身含深色版, 故未來若要加夜間模式幾乎零成本; 但 v1 不列為需求.

## 6. 開發路徑 (建議分階段)

### Phase 0: 技術原型 (1-2 週)

驗證主要技術選擇可行, 並確認以下取決於 RudyMap 檔案內容與套件能力的未知項:

- Flutter + `mapsforge_flutter` 跑得起來, 載入 RudyMap `.map` 顯示
- 評估 298MB `.map` 的載入/捏放/旋轉效能與體感, 中文字型顯示正常
- 確認 `.map` 是否含 `name:en` (決定中英切換), 是否內嵌等高線
- 確認 `hgtmix` 解壓後可餵 `mapsforge_flutter` 做 hillshade / 海拔查詢
- GPS 定位點顯示
- 評估效能, 確認方向沒走錯

**交付**: 能在 Android 手機上看到一張可縮放旋轉的 RudyMap 地圖, 並對上述未知項給出結論.

### Phase 1: 核心可用 v0.1 (4-6 週)

- RudyMap 圖資下載與管理 (App 內下載 `.map` + DEM + 樣式, 解壓, 本地匯入備援)
- 離線地圖瀏覽, 連續捏放與旋轉
- GPS 軌跡錄製 (背景, Foreground Service)
- GPX 匯入匯出 (基礎, tracks)

**交付**: 能上山錄一條軌跡, 匯出給朋友.

### Phase 2: 路線規劃 v0.2 (4-6 週)

- GraphHopper 整合 (Flutter platform channel 包裝)
- 點按起終點 + waypoint 規劃流程
- 海拔剖面圖 (互動)
- 量測長度/爬升
- GPX/KML 完整相容 (routes + waypoints + 主流 App 測試)

**交付**: 能在 App 內規劃路線, 匯出後其他 App 認得.

### Phase 3: 圖資更新與打磨 v0.3 (2-3 週)

- 設定頁完整更新流程 (HTTP HEAD 檢查更新 + 多鏡像容錯 + 原子切換)
- 軌跡錄製時即時統計 UI
- 手套/單手模式打磨

**交付**: 可實際使用的 v1.

**整體時程估計**: 扣掉本職工作時間, 業餘晚上週末約 3-4 個月可有第一個能上山測試的版本.

## 7. 已知風險

| 風險 | 影響 | 緩解 |
|------|------|------|
| OSM 台灣山徑資料不完整 | 自動路線規劃結果可能不準 (尤其中級山) | 1. 保留手動拖曳節點微調的 fallback; 2. 鼓勵使用者貢獻回 OSM |
| Android Doze/Battery Saver 限制背景定位 | 軌跡可能中斷 | Foreground Service + 顯眼通知 + 引導使用者關閉電池最佳化 |
| 圖資體積 (底圖 ~298MB + DEM ~46MB + 路由圖 ~80MB) | 首次下載大, 佔空間 | App 內下載非預先打包; 提供 Lite (168MB) 與 hgt90 (8MB) 選項 |
| `mapsforge_flutter` 為單人維護移植專案 | 效能/特性/長期維護風險 | Phase 0 實測 298MB `.map` 效能與捏放/旋轉/中文字型; 備案: 改包原生 mapsforge-map-android (PlatformView) 或 VTM |
| 依賴 RudyMap 圖資內容與發布 | `.map` 缺 `name:en`/等高線, 或對方改檔名/結構/停更, 都會影響功能 | Phase 0 先驗證內容; 多鏡像容錯; 保留手動匯入備援; 中英切換視 `name:en` 有無而定 |
| GraphHopper Flutter 整合非主流路徑 | Phase 2 開發時間風險 | Phase 0 就驗證可行; 備案: Valhalla mobile 或自寫簡單路徑搜尋 |
| 山谷 GPS 多路徑反射軌跡飄移 | 錄製軌跡品質差 | Kalman filter + accuracy > 20m 丟棄 + 速度合理性檢查 |
| GPX 各 App 格式差異 | 互通性問題 | Phase 2 結束前用 5+ 主流 App 實測來回匯入匯出 |
