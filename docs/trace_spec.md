# 軌跡與路線檔格式設計 (Trace 格式)

本文件規範 Jiudge 用來儲存"規劃路線"與"實際軌跡"的單一檔案格式 (下稱
**Trace 格式**). 本文僅定義格式與設計理由, 不含程式實作.

## 1. 背景與動機

現況: 規劃路線存成單一 JSON 物件 (`PlannedRoute`: `name` / `createdAtEpochMs` /
`waypoints` / `segments`), 一條一檔, 放在 `Documents/Jiudge/plans/`. 軌跡錄製尚未實作.

重新設計的觸發點是"可以 append"這項需求. 背景軌跡錄製在 8 小時, 約 1 Hz 取樣下會累積
近 2.8 萬個定位點. 若沿用"單一 JSON 物件", 每收到一個 GPS 點就得整檔重寫一次, 既浪費
I/O, 也讓"錄製途中閃退"直接毀掉整個檔案. 這個限制把格式直接導向**行導向 (line-oriented)**
設計.

## 2. 設計目標

對應原始需求, 逐項列出格式必須滿足的性質:

1. **共用且可辨別** - 規劃路線與實際軌跡使用同一套格式, 但能明確區分兩者.
2. **表現行進方向** - 從檔案內容可還原前進方向, 供地圖渲染箭頭.
3. **可 append** - 錄製時能逐筆附加新定位點, 不需重寫整檔, 且能容忍中途中斷.
4. **可匯出** - 能轉成 GPX 與其他常見格式 (KML, GeoJSON).
5. **可擴充** - 未來新增內容 (例如 milestone, 相片位置) 不破壞既有檔案與舊版讀取器.

## 3. 格式總覽

Trace 檔是 **JSON Lines (JSONL)**: 一個檔由多行組成, 每行是一個獨立, 完整的 JSON 物件,
以換行符 (`\n`) 分隔.

```
<header 行>
<記錄行 1>
<記錄行 2>
...
```

- **第一行**永遠是 header, 描述整個檔案.
- **其後每一行**是一筆記錄, 以欄位 `k` (kind) 判別型別.
- 編碼 UTF-8, 每行不含跨行內容, 一行對應一個邏輯單元.

行導向設計同時滿足 append (附加一行即可) 與防損毀 (逐行解析, 撕裂的末行直接略過).

## 4. Header

第一行, 描述整個檔案. 範例:

```json
{"v":1,"type":"plan","name":"大霸北稜","createdAt":1687123456789,"app":"jiudge"}
```

| 欄位 | 型別 | 必填 | 說明 |
|---|---|---|---|
| `v` | int | 是 | Trace 格式 schema 版本, 整數, 自 1 起算, 目前為 `1` |
| `type` | string | 是 | `"plan"` (規劃路線) 或 `"track"` (實際軌跡) |
| `name` | string | 是 | 顯示名稱 |
| `createdAt` | int | 是 | 建立時間, Unix epoch 毫秒 |
| `app` | string | 否 | 產生此檔的應用識別, 固定 `"jiudge"` |
| `appVersion` | string | 否 | 產生此檔的應用版本, 便於日後除錯 |

`type` 是"共用且可辨別"的核心: 兩種資料共用同一套行格式與讀寫管線, 僅以 header `type`
區分語意.

## 5. 記錄型別

每筆記錄行以 `k` 判別型別. 未知的 `k` 一律由讀取器略過 (見 §10 擴充策略). 座標一律為十進位度
(WGS84), 經緯度建議保留 7 位小數.

### 5.1 `wpt` - 使用者落點 (規劃路線)

使用者在規劃時親手放置的點 (起點, 終點, 途經點).

```json
{"k":"wpt","i":0,"lat":24.5123456,"lon":121.2345678,"name":"登山口"}
```

| 欄位 | 型別 | 必填 | 說明 |
|---|---|---|---|
| `k` | string | 是 | 固定 `"wpt"` |
| `i` | int | 是 | 落點序號, 由 0 起算, 決定先後順序 |
| `lat` / `lon` | number | 是 | 座標 |
| `name` | string | 否 | 落點名稱 |

### 5.2 `seg` - 路由幾何段 (規劃路線)

BRouter 為"第 `i` 個落點到第 `i+1` 個落點"算出的完整路徑幾何. 沿用既有
`PlannedRoute` 的設計: `seg` 是 source of truth, 一段一行, 保留逐段編輯能力; 攤平後的整條
polyline 由讀取端按需衍生.

```json
{"k":"seg","i":0,"pts":[[24.5123456,121.2345678],[24.5140000,121.2360000],[24.5234000,121.2456000]]}
```

| 欄位 | 型別 | 必填 | 說明 |
|---|---|---|---|
| `k` | string | 是 | 固定 `"seg"` |
| `i` | int | 是 | 段序號, 對應 `wpt[i]` 到 `wpt[i+1]` |
| `pts` | array | 是 | `[[lat,lon], ...]` 座標序列 |

### 5.3 `tpt` - GPS 定位點 (實際軌跡)

錄製時每一筆 GPS 取樣. 這是 append 的主要對象.

```json
{"k":"tpt","t":1687123460000,"lat":24.5123456,"lon":121.2345678,"ele":1820.4,"acc":4.0,"spd":1.1,"brg":135.0}
```

| 欄位 | 型別 | 必填 | 說明 |
|---|---|---|---|
| `k` | string | 是 | 固定 `"tpt"` |
| `t` | int | 是 | 取樣時間, Unix epoch 毫秒 |
| `lat` / `lon` | number | 是 | 座標 |
| `ele` | number | 否 | 海拔, 公尺. 僅承接匯入的外部 GPX 高度; App 自身錄製不寫 (見 §8 高度與坡度) |
| `acc` | number | 否 | 水平精度, 公尺 |
| `spd` | number | 否 | 速度, 公尺/秒 |
| `brg` | number | 否 | 方位角, 0-360 度 |

### 5.4 未來型別 (預留, 暫不實作)

擴充時只是新增一種 `k`, 不改動既有型別. 例:

```json
{"k":"mile","t":1687123900000,"lat":24.52,"lon":121.25,"label":"3K 里程牌"}
{"k":"photo","t":1687124000000,"lat":24.53,"lon":121.26,"uri":"photos/IMG_0001.jpg"}
```

這些型別尚未納入 v1 範圍, 此處僅示意擴充方式.

## 6. plan 與 track 的檔案組成

**規劃路線 (`type:"plan"`)**: 1 行 header + N 行 `wpt` + N 行 `seg`.

```
{"v":1,"type":"plan","name":"大霸北稜","createdAt":1687123456789,"app":"jiudge"}
{"k":"wpt","i":0,"lat":24.5123456,"lon":121.2345678,"name":"登山口"}
{"k":"wpt","i":1,"lat":24.5345000,"lon":121.2567000,"name":"三角點"}
{"k":"seg","i":0,"pts":[[24.5123456,121.2345678],[24.5234000,121.2456000],[24.5345000,121.2567000]]}
```

**實際軌跡 (`type:"track"`)**: 1 行 header + 多行 `tpt` (錄製過程逐行附加).

```
{"v":1,"type":"track","name":"2026-06-24 大霸","createdAt":1687123456789,"app":"jiudge"}
{"k":"tpt","t":1687123460000,"lat":24.5123456,"lon":121.2345678,"ele":1820.4,"acc":4.0}
{"k":"tpt","t":1687123461000,"lat":24.5123500,"lon":121.2345700,"ele":1820.6,"acc":3.8}
```

## 7. 行進方向

格式不另設"方向"欄位, 而是讓方向可由內容唯一還原:

- **規劃路線**: 依 `wpt` 的 `i` 與 `seg` 的 `i` 遞增順序, 即為前進方向.
- **實際軌跡**: 依 `tpt` 的時間戳 `t` 遞增順序, 即為前進方向.

渲染端據此沿線繪製方向箭頭. 如此可避免"方向"與"點序"兩個來源不一致的問題.

## 8. 高度與坡度

- **座標 (lat, lon) 是唯一的原始位置輸入**; 高度視為可由座標 + DEM 重建的衍生資料.
- App 在裝置上恆持有 RudyMap DEM (`.hgt`), 故海拔剖面與坡度一律於使用時 (渲染/量測) 以
  (lat, lon) 即時查 DEM 取得, 不寫入 trace 檔.
- 不以 GPS 回報高度算坡度: GPS 垂直精度差且抖動, 而坡度是高度對距離的微分, 會放大此雜訊;
  DEM 沿線連續平滑, 微分後坡度穩定.
- 不在檔案內存 DEM 取得的高度: 既可隨時重建, 又能避免 DEM 日後更新後存值變成過時死資料.
  如需避免重複查詢, 於載入時在記憶體快取即可, 不落地.
- `tpt.ele` (見 §5.3) 維持 optional, 僅用來承接匯入的外部 GPX 高度; App 自身錄製不寫此欄.
- 匯出 GPX 的 `<ele>` 一律改由 DEM 取得 (見 §11 匯出), 以與上述單一高度來源一致.

## 9. Append 與防損毀

- 錄製開始時建立檔案並寫入 header 行 (僅一次).
- 之後每收到一筆 GPS 取樣, 以 append 模式補一行 `tpt`, 不重寫既有內容.
- 讀取時逐行解析, 對每一行單獨 try/catch: 因閃退或電力耗盡造成的不完整末行直接略過,
  其餘記錄完整保留.
- header 與每筆記錄都是獨立的一行, 不存在"整檔需一致才可解析"的限制, 因此天然耐中斷.

### 寫入策略 (記憶體 O(1))

- foreground service 持有一個開著的 buffered 寫入器 (例如 append 模式的 `BufferedWriter`),
  每收到一筆 `tpt` 即 append 一行, 寫完即釋放該筆; **記憶體不隨軌跡長度成長, 為 O(1)**.
  嚴禁 "累積到錄製結束才一次寫出" - 那會把記憶體用量變成 O(n), 且中途中斷時全部遺失.
- 落盤分三層, 不可混為一談: (1) `BufferedWriter` 的 JVM buffer; (2) `write()` syscall 把資料
  送入 OS page cache (kernel RAM); (3) kernel 背景 writeback 把 dirty page 批次寫入 flash
  (Linux 預設約 30 秒內), 或 `fsync()` 強制立刻寫. 耗電與磨損 flash 的是第 3 層的 `fsync`,
  不是 `flush`.
- 每筆 `tpt` 就 `write()` (經 buffered writer) 即可: 資料進 page cache 後, **app 閃退不會丟**
  (kernel 持有並會回寫), 故抗閃退不需 `fsync`. `flush()` 只是把 JVM buffer 推進 page cache,
  成本極低. 批次落盤交給 OS writeback, 不需自己拉長週期省電.
- **不要每筆 (或每秒) `fsync`** - 那會強制實體寫入而傷電/磨損. 僅在自然停頓點做 `fsync`
  checkpoint (錄製暫停, app 進背景, service `onDestroy`, 或每隔幾分鐘一次).
- 唯一會遺失資料的情境是整機斷電/kernel panic, 損失上限為"上次 writeback 或 checkpoint 以來"
  的數十秒點數; 手機罕見此情況, 對軌跡錄製可接受.
- 若 UI 需邊走邊在地圖上畫軌跡而把點留在記憶體, 其量亦有限 (見下), 必要時渲染端再抽稀.

### 大小與記憶體預算

- 1 Hz 連續 12 小時 = 43,200 筆 `tpt`, 每行約 64-100 B, 檔案約 **3-4 MB**; 24 小時約 7-8 MB.
  相對於底圖等資料 (~298 MB) 屬零頭, 不需特別處理.
- 全部點即使常駐記憶體 (供即時繪製) 也僅約 2-5 MB, 遠低於 app 可用 heap, 不會爆.
- 若量級顯著放大 (多日連續或更高取樣率), 兩個不動格式的調節桿: 座標精度降為 6 位小數
  (~11 cm), 或錄製結束時將 `.jsonl` gzip 封存.

## 10. 版本與擴充策略

- **header `v`**: 不相容的結構性變更才提升 (例如座標表示法改變).
- **新增記錄型別**: 直接新增一種 `k` 即可, 不需提升 `v`.
- **向前相容**: 讀取器遇到未知 `k` 一律略過該行, 不視為錯誤. 如此舊版 App 能讀新版檔
  (忽略尚不認識的內容), 新版 App 也能讀舊版檔.
- **新增可選欄位**: 既有型別新增可選欄位時, 舊讀取器忽略未知欄位即可, 同樣不需提升 `v`.

## 11. 匯出

讀取後在記憶體中建立中介結構, 再轉出各格式.

### GPX 對應

| Trace | GPX |
|---|---|
| `type:"plan"` 的 `seg` 幾何 | `<rte>` / `<rtept>` (或攤平為單一 route) |
| `type:"plan"` 的 `wpt` | `<wpt>` (獨立航點) |
| `type:"track"` 的 `tpt` | `<trk>` / `<trkseg>` / `<trkpt>`, 含 `<ele>` 與 `<time>` |

`tpt` 的 `t` 轉為 GPX `<trkpt>` 的 `<time>` (ISO 8601 UTC); `<trkpt>` 的 `<ele>` 由 DEM 以
(lat, lon) 取得, 不採 `tpt.ele` (見 §8 高度與坡度).

### 其他格式

KML, GeoJSON 透過同一中介結構轉出. 本文件不展開細節, 僅確認格式本身保留了足夠資訊
(座標, 順序, 時間, 海拔) 以無損產生上述格式.

## 12. 儲存位置與命名

沿用既有規劃路線的儲存策略 (見 CLAUDE.md storage rationale):

- 規劃路線: `Documents/Jiudge/plans/`
- 實際軌跡: `Documents/Jiudge/tracks/`
- 副檔名: `.jsonl` (名實相符, 並讓啟動時遷移得以副檔名區分新舊格式, 見 §13).
- 命名沿用既有規則 `{slug(name)}-{createdAt}.jsonl`.
- 遷移狀態 metadata 檔: `Documents/Jiudge/.trace_meta.json` (見 §13).

## 13. 啟動時遷移

把舊版存檔一次性升級到最新 schema, 透過一個 metadata 檔記錄完工狀態, 避免每次啟動重複掃描.

### metadata 檔

- 路徑: `Documents/Jiudge/.trace_meta.json`, 一檔涵蓋 `plans/` 與 `tracks/` 兩個子目錄.
- 內容: `{"schema": <已完成升級到的版本>}`, 例如 `{"schema":1}`.
- 語意: `schema` 代表"目錄下所有 trace 都已升級到此版本".

### 流程

App 啟動且取得儲存權限後, 於背景執行:

1. 讀取 `.trace_meta.json`. 若存在且其 `schema` 不小於目前最新版本 (1), 視為已是最新,
   不做任何處理, 結束.
2. 否則掃描 `plans/` 與 `tracks/` 下所有舊格式檔 (副檔名 `.json` 的單一 JSON 物件).
3. 逐一將每個舊檔轉為新格式檔 (`.jsonl`), 檔名沿用 `{slug(name)}-{createdAt}.jsonl`.
4. 新檔成功寫出後, 刪除對應的舊 `.json`.
5. 全部處理完畢, 寫入 `.trace_meta.json` 標記 `{"schema":1}`.

### 可重入性

- 轉換以"先寫成功新檔, 再刪舊檔"為序; metadata 檔只在全部完成後才寫入.
- 若步驟 2-4 中途中斷 (例如閃退), metadata 檔尚未寫入, 下次啟動會重新掃描. 此時最多殘留
  尚未刪除的舊 `.json`, 不會遺失資料; 已轉換者 (舊 `.json` 已不存在) 自然不再被掃到.
- 因新格式用 `.jsonl`, 舊格式用 `.json`, 掃描可單憑副檔名區分新舊, 不需讀檔內容判斷.

### 未來版本升級

日後若 schema 提升到 2 以上, 沿用同一機制: 啟動時 `.trace_meta.json` 的 `schema` 小於最新版本
即觸發掃描, 依需要逐版轉換 (`.jsonl` 內容升級), 完成後更新 metadata 的 `schema`.

## 14. 與既有程式碼的對應 (概要)

非實作規範, 僅記錄預期落點, 供後續開發參考:

- 一個共用的低階逐行讀寫層, 負責 header 與記錄行的序列化/反序列化.
- 其上掛兩個領域模型: 既有 `PlannedRoute` (沿用 `segments` 為 source of truth 的編輯模型),
  與新的 `RecordedTrack`.
- 一個 `GpxExporter` (及未來其他匯出器) 讀取中介結構轉出對應格式.
- 背景軌跡錄製的 foreground service 為獨立工作項目, 不在本格式設計範圍內.
